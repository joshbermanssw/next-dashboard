# TanStack in this app ŌĆö a working guide

A grounded walkthrough of how **TanStack Router** and **TanStack Query** are used here, with every example pulled from real files in this repo. Written to be read offline. No prior TanStack experience assumed.

---

## 1. Mental model

There are two TanStack libraries doing different jobs:

| Library              | Owns                                                                                     | Compare to                |
|----------------------|------------------------------------------------------------------------------------------|---------------------------|
| **`@tanstack/react-router`** | URL ŌåÆ component tree, route guards, navigation, params/search.                  | Next.js App Router        |
| **`@tanstack/react-query`**  | Async server state ŌĆö fetching, caching, deduping, refetching, mutations.        | Next's `fetch` cache, SWR |

They are independent but compose well: **the router calls Query inside `beforeLoad`** to gate access to a route, and **components call Query inside the route's `component`** to read the same data without re-fetching it.

The single shared object between them is the `QueryClient`. We pass it into the router as `context` so route loaders can access it.

## 2. Bootstrapping (`src/main.tsx`)

This is where both libraries are wired together. Read the file alongside this section.

```ts
// src/main.tsx
const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      staleTime: 30_000,      // 30s before a query is considered stale
      retry: 1,               // retry failed queries once
      refetchOnWindowFocus: false,
    },
  },
})

const router = createRouter({
  routeTree,                  // generated by the Vite plugin
  context: { queryClient },   // available to every route's beforeLoad / loader
  defaultPreload: "intent",   // prefetch route data when user hovers a <Link>
})

// Tell TypeScript what router shape we registered, so useNavigate etc. are typed.
declare module "@tanstack/react-router" {
  interface Register { router: typeof router }
}
```

Then we wrap the app:

```tsx
<QueryClientProvider client={queryClient}>
  <TooltipProvider>
    <RouterProvider router={router} />
  </TooltipProvider>
</QueryClientProvider>
```

Order matters: `QueryClientProvider` must be *above* `RouterProvider` because route components call `useQuery` etc., and those hooks need the provider in scope.

### Key options to know

- **`staleTime`** ŌĆö how long a cached value is considered "fresh" (no refetch on remount or window focus). Default is `0`, which is aggressive. We bumped it to 30s.
- **`retry`** ŌĆö retry failed queries N times before surfacing the error. Set per-query or globally.
- **`defaultPreload: "intent"`** ŌĆö when the user hovers a `<Link>`, the router prefetches the route's loaders / `beforeLoad`. Feels instant on click. Set to `false` or `"viewport"` to change behavior.

## 3. TanStack Router

### 3.1 File-based routing

Routes live in `src/routes/`. The **`@tanstack/router-plugin/vite`** plugin watches that directory and generates `src/routeTree.gen.ts` (gitignored). The mapping from filename ŌåÆ URL:

| File                                  | URL                | What it is                                      |
|---------------------------------------|--------------------|-------------------------------------------------|
| `src/routes/__root.tsx`               | (none)             | The root route. Always rendered. Holds context. |
| `src/routes/login.tsx`                | `/login`           | Top-level route.                                |
| `src/routes/_dashboard.tsx`           | (none ŌĆö pathless)  | A **layout route**. Its `Outlet` wraps children, but it adds no segment to the URL. |
| `src/routes/_dashboard.index.tsx`     | `/`                | The index inside the `_dashboard` layout.       |
| `src/routes/posts.$postId.tsx` (hypothetical) | `/posts/:postId` | Dynamic segment.                       |

Naming rules to memorize:
- **Leading `_`** ŌĆö pathless layout. The folder/file contributes a layout but **not** a URL segment. We use `_dashboard` to wrap the sidebar + header + auth guard around `/`, `/accounts`, etc., without adding a `/dashboard` prefix.
- **`$param`** ŌĆö dynamic segment. Becomes `params.param` inside `loader` / `component`.
- **`.`** as separator in flat-file mode joins segments: `_dashboard.index.tsx` is the index child of `_dashboard`. You could also use folders (`_dashboard/index.tsx`) ŌĆö flat-file is what we use.
- **`__double-underscore`** ŌĆö never part of the URL; reserved for special files like `__root.tsx`.

### 3.2 Anatomy of a route file

Every route file exports a `Route` from `createFileRoute(path)`. The path string passed in is a **type-level hint** that lets TanStack verify the filename matches.

```tsx
// src/routes/_dashboard.index.tsx
export const Route = createFileRoute("/_dashboard/")({
  component: DashboardHome,
})
```

You can attach any of these to a route:

| Field         | When it runs                                          | What it can do                                                   |
|---------------|-------------------------------------------------------|------------------------------------------------------------------|
| `beforeLoad`  | Before the route mounts; runs top-down through parents | Auth guards, side effects, populate context. Can `throw redirect()`. |
| `loader`      | After `beforeLoad`. Result is cached by the router.   | Fetch data the component will block on. Often delegates to Query. |
| `component`   | Renders when load is complete.                        | The page UI.                                                     |
| `errorComponent` | When `loader`/`beforeLoad` throws.                 | Custom error UI.                                                 |
| `pendingComponent` | While loading (with a `pendingMs` threshold).    | Custom loading UI.                                               |
| `validateSearch` | On every navigation.                              | Parse + type the `?search` query (often with Zod).               |

In this repo we mostly use `beforeLoad` + `component`. We let TanStack **Query** own the actual data fetching, so `loader` is unused ŌĆö we delegate to `queryClient.ensureQueryData` from `beforeLoad`.

### 3.3 Root route + router context (`src/routes/__root.tsx`)

```tsx
type RouterContext = { queryClient: QueryClient }

export const Route = createRootRouteWithContext<RouterContext>()({
  component: () => <Outlet />,
})
```

Two things are happening:

1. `createRootRouteWithContext<RouterContext>()` is a **factory that returns the real route constructor**, but with the context type baked in. Required so `context.queryClient` is typed inside every descendant's `beforeLoad`.
2. The component is just `<Outlet />`. Whatever child route is active renders here. Put global chrome (devtools, top-level providers) at this layer if you need it.

The `context` object you declared here is what you passed into `createRouter({ context: { queryClient } })` in `main.tsx`. The values flow top-down: descendant routes can also *add* to the context inside their own `beforeLoad` by returning an object (advanced ŌĆö not used here).

### 3.4 Pathless layout: `_dashboard.tsx`

```tsx
export const Route = createFileRoute("/_dashboard")({
  beforeLoad: async ({ context, location }) => {
    const me = await context.queryClient.ensureQueryData(meQueryOptions)
    if (!me) {
      throw redirect({ to: "/login", search: { redirect: location.href } })
    }
  },
  component: DashboardLayout,
})
```

This file is doing several things at once:

- **The path `/_dashboard`** means the route is pathless ŌĆö see the leading `_`. It contributes the sidebar + header layout but no URL segment.
- **`beforeLoad`** receives `{ context, location, params, search, ... }`. It calls into the QueryClient via `context.queryClient.ensureQueryData(meQueryOptions)`.
  - `ensureQueryData(options)` returns the cached value if it exists and is fresh; otherwise it runs the query function and caches the result. It **does not** subscribe ŌĆö it's a one-shot read for guards/loaders.
- **`throw redirect(...)`** ŌĆö this is how you redirect in TanStack Router. You *throw*; the router catches it and navigates. Returning a value from `beforeLoad` does nothing. Throw `redirect(...)` or throw `notFound()`.
- **`search: { redirect: location.href }`** ŌĆö passes the original URL as a search param so the login page could return the user where they came from. (We don't currently consume it on the login page, but the wiring is there.)
- **`component: DashboardLayout`** ŌĆö renders the sidebar, header, and an `<Outlet />`. Any matched child (`_dashboard.index.tsx`, etc.) renders into that outlet.

Inside the component, the same query is read with `useSuspenseQuery`:

```tsx
function DashboardLayout() {
  const { data: customer } = useSuspenseQuery(meQueryOptions)
  // ...
}
```

Because `ensureQueryData` already populated the cache during `beforeLoad`, `useSuspenseQuery` resolves immediately ŌĆö no flash of spinner.

### 3.5 The login guard (inverse case)

`src/routes/login.tsx`:

```tsx
export const Route = createFileRoute("/login")({
  beforeLoad: async ({ context }) => {
    const me = await context.queryClient.ensureQueryData(meQueryOptions)
    if (me) throw redirect({ to: "/" })
  },
  component: LoginPage,
})
```

Same pattern, opposite direction: if the user is already authenticated, kick them to the dashboard.

### 3.6 Navigation primitives

| API                                  | Where to use it                                                                 |
|--------------------------------------|---------------------------------------------------------------------------------|
| `<Link to="/accounts">`              | Anchor tags. Use everywhere instead of `<a href>` for client-side nav.          |
| `useNavigate()`                      | Programmatic navigation. `navigate({ to: "/login" })`.                          |
| `useLocation()`                      | Current URL, pathname, search. Used in `app-sidebar.tsx` to highlight active nav.|
| `useParams({ from: "..." })`         | Typed access to dynamic segments.                                               |
| `useSearch({ from: "..." })`         | Typed access to validated search params.                                        |
| `throw redirect({ to })`             | Inside `beforeLoad`/`loader`/mutations ŌĆö synchronous-ish redirect.              |

Real examples in this codebase:

```tsx
// src/components/app-sidebar.tsx
const { pathname } = useLocation()
// ...
render={<Link to={item.url} />}    // hands the <a> rendering to TanStack Link

// src/components/nav-user.tsx
const navigate = useNavigate()
navigate({ to: "/login" })

// src/routes/login.tsx ŌĆö inside mutation onSuccess
navigate({ to: "/" })
```

### 3.7 `defaultPreload: "intent"`

Because we set this on the router, every `<Link>` automatically prefetches its target's `beforeLoad` and `loader` on hover/focus. For the dashboard nav, that means by the time the user clicks "Accounts", the route's auth check has likely already run and any `ensureQueryData` calls have populated the cache. You get this for free.

### 3.8 Search params (when you need them)

To use typed search params in a route:

```tsx
import * as z from "zod"

const SearchSchema = z.object({
  page: z.number().int().min(1).catch(1),
  q: z.string().optional(),
})

export const Route = createFileRoute("/posts")({
  validateSearch: SearchSchema,
  loaderDeps: ({ search }) => ({ page: search.page, q: search.q }),
  loader: ({ deps }) => fetchPosts(deps),
  component: () => {
    const { page, q } = Route.useSearch()
    return <PostsList page={page} q={q} />
  },
})
```

We don't use this yet, but `validateSearch` is the idiomatic way to parse `?page=2&q=...`. `loaderDeps` tells the router which search values invalidate the loader cache.

## 4. TanStack Query

Query owns server state: fetching, caching, dedup, refetch, mutations.

### 4.1 `queryOptions` ŌĆö the building block

`queryOptions` is a small helper that lets you define a query once and reuse the same definition everywhere:

```ts
// src/queries/user.ts
export const meQueryOptions = queryOptions({
  queryKey: ["me"],
  queryFn: fetchMe,
  staleTime: 5 * 60_000,   // 5 minutes
  retry: false,
})
```

- **`queryKey`** ŌĆö the cache key. An array; serialized for cache lookup. Add inputs to the array if the data varies (e.g., `["user", userId]`).
- **`queryFn`** ŌĆö async function that returns the data, or throws on error. Signature: `({ queryKey, signal, ... }) => Promise<T>`.
- **`staleTime`** ŌĆö override per query. Here we say `/me` is fresh for 5 minutes.
- **`retry: false`** ŌĆö `/me` returning 401 is *normal* (not authenticated). We don't want to retry that.

The benefit of `queryOptions` over inlining: every consumer references the same key/function, and TypeScript infers a consistent type. Use it in `useQuery`, `useSuspenseQuery`, `ensureQueryData`, `prefetchQuery`, `invalidateQueries({ queryKey: meQueryOptions.queryKey })`, etc.

### 4.2 The three ways to read a query

| API                  | Behavior                                                                                | When to use                                                |
|----------------------|-----------------------------------------------------------------------------------------|------------------------------------------------------------|
| `useQuery(opts)`     | Subscribes. Returns `{ data, isPending, isError, error, ... }`. Component renders during loading. | When you want to handle loading/error states inline.       |
| `useSuspenseQuery(opts)` | Subscribes. **Suspends** the component until data is ready. `data` is non-nullable.  | When the route's `beforeLoad`/`loader` already ensured data. |
| `queryClient.ensureQueryData(opts)` | Imperative. Returns a promise that resolves to data; runs `queryFn` if not cached. **Does not** subscribe. | Inside `beforeLoad`/`loader`/event handlers.               |

In this app:

```tsx
// route guard ŌĆö imperative, doesn't subscribe
await context.queryClient.ensureQueryData(meQueryOptions)

// component ŌĆö subscribes, suspends, data is guaranteed
const { data: customer } = useSuspenseQuery(meQueryOptions)
```

That pairing is why the dashboard never shows a "Loading customer..." spinner ŌĆö by the time the component mounts, `meQueryOptions` has already resolved.

### 4.3 Caching, staleness, and invalidation

A query has two relevant time concepts:

- **`staleTime`** ŌĆö how long data is "fresh". While fresh, no refetch happens on remount, focus, or reconnect.
- **`gcTime`** (default 5 min) ŌĆö how long the cache entry survives after the *last* subscriber unmounts. After this, it's garbage-collected.

To force a refetch:

```ts
// Mark every query whose key starts with ["me"] as stale; refetches all active ones.
queryClient.invalidateQueries({ queryKey: ["me"] })

// Or directly set the cached value (used in this app for logout):
queryClient.setQueryData(["me"], null)

// Or refetch a specific query without invalidating:
queryClient.refetchQueries({ queryKey: ["me"] })
```

We do the `setQueryData(["me"], null)` trick in `nav-user.tsx` after logout ŌĆö it's an optimistic local update so the UI doesn't need to wait for `/me` to fail. The next route navigation's `beforeLoad` reads the cache and redirects to `/login`.

### 4.4 Mutations

For writes (login, logout, anything that changes server state), use `useMutation`:

```tsx
// src/routes/login.tsx
const mutation = useMutation({
  mutationFn: login,                         // async fn from src/api/auth.ts
  onSuccess: () => {
    setState(undefined)
    navigate({ to: "/" })
  },
  onError: (err) => {
    if (err instanceof BffError) {
      setState(err.payload as LoginFormState ?? { message: err.message })
    } else {
      setState({ message: "Unable to connect to authentication service." })
    }
  },
})

// trigger:
mutation.mutate(values)
// or, if you want to await:
await mutation.mutateAsync(values)
```

State you get back:

- `mutation.isPending` ŌĆö request in flight (used to disable the button + show spinner)
- `mutation.isSuccess`, `mutation.isError`
- `mutation.data`, `mutation.error`
- `mutation.reset()`

After a mutation, you typically invalidate related queries:

```ts
onSuccess: () => {
  queryClient.invalidateQueries({ queryKey: ["accounts"] })
}
```

We don't do this in `login` because we don't have an existing `/me` cache to invalidate ŌĆö but if we wanted the dashboard to immediately read the new customer, we'd `queryClient.setQueryData(meQueryOptions.queryKey, response.payload.customer)` inside `onSuccess`.

### 4.5 Optimistic updates (cookbook)

```ts
const updateName = useMutation({
  mutationFn: api.updateName,
  onMutate: async (newName) => {
    await queryClient.cancelQueries({ queryKey: ["me"] })
    const previous = queryClient.getQueryData<Customer>(["me"])
    queryClient.setQueryData<Customer>(["me"], (old) =>
      old ? { ...old, firstName: newName } : old
    )
    return { previous }                                 // context for onError
  },
  onError: (_err, _vars, context) => {
    queryClient.setQueryData(["me"], context?.previous) // rollback
  },
  onSettled: () => {
    queryClient.invalidateQueries({ queryKey: ["me"] })
  },
})
```

This is the standard 4-step optimistic update pattern: cancel in-flight, snapshot, mutate, rollback on error, refetch on settle.

## 5. End-to-end walkthrough: the auth flow in this app

Step through what happens when an unauthenticated user opens the app at `/`.

1. **Vite serves `index.html`.** It loads `/src/main.tsx`.
2. **`main.tsx`** instantiates `QueryClient`, instantiates `router` with `context: { queryClient }`, and renders `<RouterProvider>`.
3. **Router resolves `/`** to `_dashboard.tsx` ŌåÆ `_dashboard.index.tsx` (because `_dashboard` is pathless and `index` is its child).
4. **`_dashboard.tsx`'s `beforeLoad` runs first** (parents before children).
   - Calls `context.queryClient.ensureQueryData(meQueryOptions)`.
   - The query is not in cache, so `fetchMe` runs.
   - `fetchMe` does `fetch("/api/bff/auth/me", { credentials: "include" })`.
   - Vite dev server proxies `/api/bff/*` to the Hono BFF on `:8787`.
   - BFF's `requireSession` middleware reads the `session` cookie, finds none, returns 401.
   - `fetchMe` catches `BffError` with `status === 401` and returns `null`.
   - `ensureQueryData` resolves with `null` and caches it.
   - `beforeLoad` sees `me === null` and throws `redirect({ to: "/login" })`.
5. **Router catches the redirect**, navigates to `/login`.
6. **`/login`'s `beforeLoad` runs**: `ensureQueryData(meQueryOptions)` returns the cached `null` ŌĆö no second fetch. Falsy, so no redirect.
7. **`LoginPage` renders.** User submits the form.
8. **`useMutation`'s `mutationFn` calls `login(values)`** which posts to `/api/bff/auth/login`.
9. **BFF validates with Zod, proxies to `MS_AUTH_BASE_URL/auth/login`,** signs a JWT with Jose, sets the `httpOnly` `session` cookie via `Set-Cookie`.
10. **Mutation's `onSuccess`** calls `navigate({ to: "/" })`.
11. **Router runs `_dashboard.tsx`'s `beforeLoad` again.** `meQueryOptions` is still cached as `null` (staleTime 5 min) ŌĆö but **we didn't invalidate it after login**. This is actually a bug worth flagging: we should `queryClient.setQueryData(meQueryOptions.queryKey, customer)` in `onSuccess` so the post-login navigation sees the fresh value.

    (Currently the flow works only because route navigation re-evaluates `beforeLoad` and, depending on timing, `ensureQueryData` may refetch ŌĆö but the safe pattern is to write the cache explicitly. Left as an exercise / TODO.)

12. **`useSuspenseQuery(meQueryOptions)` in `DashboardLayout`** reads the (now populated) customer; sidebar + header + dashboard home render.

### Logout (`nav-user.tsx`)

1. User clicks "Log out".
2. `handleLogout` calls `logoutRequest()` ŌåÆ `POST /api/bff/auth/logout`.
3. BFF deletes the cookie, best-effort notifies upstream.
4. `queryClient.setQueryData(["me"], null)` ŌĆö local cache becomes "not authenticated" instantly.
5. `navigate({ to: "/login" })`.
6. The login route's `beforeLoad` reads the now-`null` `me`, doesn't redirect, the form renders.

## 6. Cookbook ŌĆö common operations you'll want

### Add a new authenticated page (e.g. `/accounts`)

1. Create `src/routes/_dashboard.accounts.tsx`:

   ```tsx
   import { createFileRoute } from "@tanstack/react-router"
   export const Route = createFileRoute("/_dashboard/accounts")({
     component: AccountsPage,
   })
   function AccountsPage() {
     return <div className="px-4 py-6 lg:px-6">Accounts go here</div>
   }
   ```

2. That's it ŌĆö the auth guard is inherited from `_dashboard`, the sidebar/header come from the layout, and the link in `app-sidebar.tsx` already exists for `/accounts`. The Vite plugin regenerates `routeTree.gen.ts` on save.

### Add a new data query

1. Define options:

   ```ts
   // src/queries/accounts.ts
   import { queryOptions } from "@tanstack/react-query"
   import { fetchAccounts } from "@/api/accounts"

   export const accountsQueryOptions = queryOptions({
     queryKey: ["accounts"],
     queryFn: fetchAccounts,
     staleTime: 60_000,
   })
   ```

2. Optionally prefetch in `beforeLoad` so the page is instant:

   ```tsx
   beforeLoad: ({ context }) =>
     context.queryClient.prefetchQuery(accountsQueryOptions),
   ```

   Use `prefetchQuery` (fire-and-forget) when you don't want to block route entry. Use `ensureQueryData` when you must block.

3. Read in the component with `useSuspenseQuery` (if prefetched) or `useQuery` (if you want to render loading state).

### Parameterized queries

```ts
export const accountQueryOptions = (id: string) =>
  queryOptions({
    queryKey: ["account", id],
    queryFn: () => fetchAccount(id),
  })

// usage
const { data } = useSuspenseQuery(accountQueryOptions(params.id))
```

Each `id` gets its own cache entry.

### Aborting in-flight requests

`queryFn` receives an `AbortSignal` you can pass to `fetch`. The router cancels signals when you navigate away:

```ts
queryFn: ({ signal }) => fetch("/api/bff/things", { signal }).then((r) => r.json())
```

### Devtools

Add to `main.tsx` for the router (and there's an equivalent `@tanstack/react-query-devtools`):

```tsx
import { TanStackRouterDevtools } from "@tanstack/router-devtools"
// then somewhere inside the providers:
<TanStackRouterDevtools router={router} />
```

`@tanstack/router-devtools` is already in `devDependencies`.

## 7. Gotchas and conventions

- **The path string in `createFileRoute("/whatever")` must match the filename.** TypeScript will yell if it doesn't. Don't try to "rename" routes by editing only the string.
- **`routeTree.gen.ts` is auto-generated** and gitignored. Don't hand-edit it. The Vite plugin regenerates it on file save during `npm run dev` and on `npm run build`. If TypeScript complains it can't find a route, run a build to regenerate.
- **Throw, don't return, redirects.** `return redirect(...)` does nothing.
- **`beforeLoad` runs parent ŌåÆ child.** Auth guards on a layout route protect all descendants automatically.
- **`useSuspenseQuery` will suspend.** Make sure something above it can handle suspense (the router does). Don't put it above a `<Suspense>` boundary that doesn't exist.
- **`credentials: "include"` on every BFF fetch.** Required for the httpOnly session cookie to ride along. It's already centralized in `src/api/auth.ts`'s `bff()` helper ŌĆö use that helper for any new BFF call.
- **Don't share `queryClient` across users.** Standard caveat: if you ever do SSR or have multiple users in the same browser context (you don't here), instantiate per-request/per-user.
- **`staleTime` vs `gcTime`** ŌĆö easy to confuse. `staleTime` controls refetch behavior for active subscribers; `gcTime` controls how long unused entries linger in memory.
- **Mutations don't auto-invalidate.** You must explicitly `invalidateQueries` or `setQueryData` in `onSuccess` / `onSettled` to keep the cache honest.

- Router docs: https://tanstack.com/router/latest/docs/framework/react/overview
- Query docs: https://tanstack.com/query/latest/docs/framework/react/overview
- The `examples/` folder of either repo on GitHub is the fastest way to learn a specific pattern (file-based routing, optimistic updates, infinite queries, etc.).

The two phrases worth burning in:

> **`ensureQueryData` in `beforeLoad`, `useSuspenseQuery` in the component.**
>
> **Throw `redirect()`, don't return it.**

Everything else is variation on those.
